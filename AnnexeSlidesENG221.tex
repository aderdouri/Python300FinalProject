\begin{frame}{\centerline{Evaluation de la vulnérabilité - AVA\_VAN}}
L'évaluation de la vulnérabilité se résume comme suit:
\begin{enumerate}
\item [D]: Le développeur doit fournir la TOE pour les tests.
\item [C]: La TOE doit être testable.
\item [E1]: L'évaluateur doit confirmer que les informations fournies satisfont à toutes les exigences en matière de contenu et de présentation des preuves.
\item [E2]: L'évaluateur doit effectuer une recherche dans les sources du domaine public afin d'identifier les vulnérabilités potentielles dans la TOE.
\end{enumerate}
\end{frame}


\begin{frame}{\centerline{Evaluation de la vulnérabilité - AVA\_VAN}}
\begin{enumerate}
\item [E3]: L'évaluateur doit effectuer une analyse \emph{indépendante/indépendante ciblée/indépendante méthodique/indépendante méthodique avancée} de la vulnérabilité de la TOE en utilisant \emph{la documentation d'orientation, les spécifications fonctionnelles, la conception, la description de l'architecture de sécurité et la représentation de l'implémentation} pour identifier les vulnérabilités potentielles dans la TOE.	
\item [E4]: L'évaluateur doit effectuer des tests de pénétration, basés sur les vulnérabilités potentielles identifiées pour déterminer si la TOE résiste aux attaques menées par un attaquant possédant un potentiel d'attaque \emph{Basique/ Basique -Amélioré/Modéré/Elevé}.
\end{enumerate}
\end{frame}

\begin{frame}{\centerline{Exemple de cotation d'une attaque - 1}}
Hypothèses:
\begin{enumerate}
\item [1)] Une TOE pourrait utiliser un simple mécanisme d'authentification par mot de passe
\item [2)] La durée d'accès à un terminal n'est pas contrôlée d'une façon efficace
\item [3)] Le système impose les restrictions suivantes sur le mot de passe :
\begin{itemize}
\item Le mot de passe est sur 4 à 6 chiffres
\item Les séquences numériques consécutives sont interdites (7-6-5-4-3)
\item Les chiffres ne doivent pas être répétés 
\item Le terminal est verrouillé pendant une heure après la sixième tentative d'authentification échouée.
\item Remise à zéro du compteur au bout de cinq minutes
\item L'attaquant puisse tenter 60 fois / heure
\end{itemize}
\end{enumerate}
\end{frame}


\begin{frame}{\centerline{Exemple de cotation d'une attaque - 2}}
Le pire des cas : Cas d'un mot de passe sur 4 chiffres:
\begin{enumerate}
\item [1)] Le nombre de permutations possible du mot de passe: \(n=\frac{10!}{(10-4)!}=5040\)
\item [2)] Le nombre de séquences croissantes et décroissantes est \(7  + 7 = 14\)
\item [3)] Le nombre de permutations possible résultant: \(n=5040-14=5026\)
\end{enumerate}
\end{frame}


\begin{frame}{\centerline{Exemple de cotation d'une attaque - 3}}
\begin{enumerate}
\item [4)] En moyenne l'attaquant doit rentrer 2513 fois un mot de passe toutes les 60 minutes
\item [5)] Le nombre de permutations possible \(\frac{2513}{60 minutes} \simeq 42 heures\) avant d'enter le bon mot de passe
\item [6)] Selon la table utilisé on obtient une note inférieur à 9\\
\(\Rightarrow\) Pas de classement
\end{enumerate}
\end{frame}

\begin{frame}{\centerline{Problèmes des CC}}
\begin{enumerate}	
	\item [1)] Evaluation statique du produit (TOE fixe)
	\item [2)] Temps d'évaluation longs par rapport à des cycles de développement de plus en plus rapides
	\item [3)] Des correctives peuvent être livrés au cours de l'évaluation\\
	\(\Rightarrow\) augmente la durée de l'évaluation
	\item [4)] Pas suffisamment connue par les développeurs commerciaux
	\item [5)] Les vulnérabilités de codage ne sont pas pris en compte
	\item [6)] Perçue comme n'améliorant pas la sécurité (littérature)	
\end{enumerate}
\end{frame}


\begin{frame}{\centerline{Avantages des CC}}
\begin{enumerate}	
\item [1)] Reconnaissance internationale
\item [2)] La maintenance assure la continuité des certificats
\item [3)] Extensibilité des SARs
\item [4)] Philosophie des CC: Nivellement des tâches d'évaluation
\begin{itemize}
	\item Périmètre de l'analyse
	\item Profondeur d'analyse
	\item Niveau de rigueur de l'évaluation		
\end{itemize}	
\item [5)] Généricité des STs via des PPs déjà approuvés
\end{enumerate}
\end{frame}
